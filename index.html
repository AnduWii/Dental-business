Ad revenue basics

Once in YPP, ads run on long-form videos and Shorts. Shorts ad revenue comes from the Shorts ad pool, shared based on your share of views, then split with you. Earnings depend on views, viewer geography, niche (CPM/RPM), and seasonality.

Setup & information

1 · Setup instructions (Google Cloud, OAuth, APIs, keys)
  1. Create a project at console.cloud.google.com.
  2. In APIs & Services → Library, enable YouTube Data API v3 and YouTube Analytics API.
  3. In OAuth consent screen: choose External, fill basics, and add your Google account under Test users (while in Testing mode). Add scopes youtube.readonly and yt-analytics.readonly.
  4. In Credentials → Create credentials → OAuth client ID: type Web application. Under Authorized JavaScript origins, add the exact origin you serve this file from (e.g. http://localhost:8000) — no path, no trailing slash. Copy the Client ID.
  5. (Optional) Create an API key — not required here because requests use your OAuth token.
  6. Open this HTML file, find PASTE_YOUR_CLIENT_ID_HERE in the CONFIG block, and paste your Client ID.
  7. Serve over http/https (not file://): python3 -m http.server 8000 then open http://localhost:8000/youtube-shorts-dashboard.html.
  8. Click Sign in with Google, grant read-only access, and your data loads.
2 · Security considerations
  • Read-only scopes only — cannot modify, upload, delete, or post.
  • No financial scope — yt-analytics-monetary.readonly is intentionally not requested; revenue tab is educational. App never touches banks/payment processors.
  • Token in memory only — never written to localStorage/cookies/servers. Closing the tab discards it; Sign out revokes it with Google.
  • Implicit flow — no refresh token client-side, tokens expire (~1h), then re-auth. Authorization-code flow needs a backend.
  • Client ID is public-safe — restricted by your Authorized JavaScript origins. Never put a client secret in client code.
  • Third-party CDNs — Google GSI + Chart.js. Self-host + add Subresource Integrity for a stricter supply chain.
3 · API limitations
  • Quota: Data API default 10,000 units/day. This app uses only a few units/refresh, but be mindful.
  • Analytics latency: data is delayed ~2–3 days; the most recent 1–2 days are often incomplete.
  • Shorts detection: no reliable is-Short API field, so classification is by duration (≤60s by default — editable in CONFIG). Edge cases may be misclassified.
  • Privacy thresholds: some Analytics dimensions (e.g. demographics) are withheld below minimum view thresholds.
  • Pagination: this demo loads up to a capped number of recent uploads (CONFIG.MAX_VIDEOS) to stay fast and within quota.
4 · Honest limitations — what the data can & cannot tell you
  • Best time to post is correlational. We see publish time + accumulated views, not a randomized experiment. It's a hint, not causation.
  • Forecasts are estimates. Virality depends on recommendation-system behavior that isn't exposed by any API. Treat probabilities as rough, confidence-tagged guesses from your own history.
  • "Audience activity patterns" here are inferred from your publish/performance history, not real-time viewer presence (which the public API doesn't provide).
  • Retention is limited to averageViewPercentage / averageViewDuration via the API; frame-by-frame retention graphs live in Studio, not the public API.
  • Health score is relative to your own channel medians — not an absolute quality measure or a Studio metric.
5 · Features that would require a backend server
  • Silent token refresh via OAuth authorization-code flow (store refresh token server-side).
  • Scheduled snapshots / historical tracking (cron jobs storing daily metrics to build true longitudinal datasets and detect anomalies).
  • True ML forecasting trained across many videos/channels, served from an API.
  • Quota pooling & caching for multiple users; hiding API keys; rate-limit protection.
  • Email/push alerts ("a Short is overperforming"), exports, and multi-channel comparison.
  • Webhooks / PubSubHubbub for near-real-time new-upload notifications.
6 · Future upgrade recommendations
  • Pull traffic sources, impressions & CTR (where available), and audience geography from Analytics for deeper "best time" + targeting insight.
  • Replace the heuristic Shorts filter with a confirmed list (e.g., cross-check via your own labeling).
  • Store time-series locally (IndexedDB) to build real per-video growth history across sessions.
  • Add A/B-style comparisons of titles/thumbnails over time, and tag experiments.
  • Backend ML model (e.g., survival / hazard models) for milestone probabilities; calibrate against outcomes.
  • Add CSV/PDF export and a shareable read-only "media kit" view for sponsorships.
  • Subresource Integrity + self-hosted libraries; optional PKCE auth-code flow with a tiny serverless function.
7 · Methodology (how each estimate is computed)

View velocity = lifetime views ÷ age in days (and ÷24 for per-hour).

Best time: each Short's views/day is bucketed by local publish weekday & hour; we average per bucket and rank. Confidence scales with the number of Shorts in a bucket.

Forecast: we model cumulative views as V∞·(1 − e^(−t/τ)) with τ≈ days. From a Short's current views at age t we estimate its lifetime total V∞, then model the final outcome as log-normal (σ from your channel's view spread, widened for very young videos) to get P(final ≥ milestone). Already-reached milestones are 100%.

Health score: weighted blend of engagement rate, like/view, comment/view, views/day, and retention — each scored by a logistic of its log-ratio to your channel median (median ⇒ 50).

Base rates: fraction of your Shorts older than 14 days that crossed each milestone.
